# Starblocks 개선 기획서

**작성일**: 2026-04-26
**범위**: UI/UX 개편 + 자소서 워크플로 보강
**목표**: "데모로 한번 써보는 도구"에서 "실제 지원 시즌에 며칠 들고 다니는 도구"로 격상

---

## 0. 현황 진단

### UI/UX 측면
- Tailwind 기본기는 갖춰져 있으나 디자인 정체성 부재 (제너릭 SaaS 룩)
- 이모지로 시각 요소 대체, 일관된 아이콘 셋 없음
- 대시보드가 평면적 — 업종 추천 / 블록 리스트 / 역량 커버리지가 동등 비중으로 나열되어 위계 없음
- 랜딩 페이지가 100% 텍스트 — 데모/시각화 없이 가치 제안을 글로만 전달

### 기능 측면 (가장 큰 빈 자리)
1. **자소서 작업 결과가 휘발됨** — `useState`로만 보유, 새로고침 시 초기화. 회사별 히스토리 관리 불가
2. **자소서 답변 재생성 불가** — 마음에 안 들면 수동 편집뿐. 톤/스타일 변주 옵션 없음
3. **자동 매칭된 블록을 사용자가 교체 불가** — 매칭 오류 시 우회로 없음

## 1. 작업 순서 (확정)

UI 개편과 백엔드 작업을 **번갈아 배치**해 컨텍스트 스위칭 비용을 줄이고, 의존성 충돌을 피한다.

| Phase | 작업 | 분류 | 예상 공수 | 의존성 |
|-------|------|------|-----------|--------|
| **P1** | 디자인 토큰 정비 (mini) | UI | 2~3h | — |
| **P2** | 자소서 저장 인프라 + 이력 페이지 | 기능 (DB) | 6~8h | — |
| **P3** | 자소서 재생성 + 톤 조정 + 블록 수동 매칭 | 기능 | 4~6h | P2 |
| **P4** | 대시보드 정보 위계 개편 | UI | 6~8h | P1 |
| **P5** | 랜딩 페이지 데모/시각화 | UI | 4~6h | P1 |
| **P6** | 보강 묶음 (검색·필터·재시도 등) | 기능 (소) | 3~4h | P4 |

### 순서를 이렇게 잡은 이유

- **P1을 가장 먼저**: 컬러·그림자·아이콘 토큰을 먼저 잡아두면 P4·P5에서 컴포넌트를 새로 만들 때 두 번 손볼 필요가 없다. 단, "완벽한 디자인 시스템" 욕심을 내면 끝이 없으므로 **컬러 토큰 / 카드 스탠다드 / lucide 아이콘 교체** 3가지로 범위를 잘라 2~3시간에 마감한다.
- **P2를 P4보다 먼저**: 자소서 저장이 가장 큰 사용자 페인이고, DB 스키마 변경을 동반하므로 UI 작업과 섞이면 디버깅이 복잡해진다. 백엔드는 백엔드끼리 묶어서 처리.
- **P3은 P2 직후**: 자소서 저장 흐름이 잡히면 "재생성 시 새 버전을 저장할 것인가"라는 질문이 자연스럽게 따라온다. P3을 P2 직후에 두면 데이터 모델 결정을 한 번에 내릴 수 있다.
- **P4 → P5**: 대시보드는 매일 보는 화면, 랜딩은 진입 1회만. 사용자 보유 자산(블록)을 보여주는 화면을 먼저 손본다.
- **P6 마지막**: 자잘한 보강은 큰 변경 후에 모아서 처리해야 회귀 테스트 부담이 적다.

## 2. Phase 별 상세

### P1. 디자인 토큰 정비 (2~3h)

**범위 (잘라낸 것)**: 컬러 / 카드 / 아이콘 3가지만. 타이포 스케일·간격 시스템·다크모드 등은 이번 범위에서 제외.

#### 작업
1. **컬러 토큰 보강** (`src/index.css`)
   - 카테고리별 액센트 컬러 토큰 추가 (10개 카테고리 → 5~6개 그룹으로 묶어서)
   - `--color-surface-*` (카드 배경/보더), `--shadow-card-*` 토큰 정의
2. **카드 컴포넌트 표준화**
   - 현재 `bg-white rounded-xl border border-slate-200` 패턴이 반복됨 → `Card` 컴포넌트 추출
   - Hover 상태 표준 (그림자 강화 + 미세 lift transform)
   - 카테고리별 좌측 컬러 바 슬롯
3. **lucide-react 도입** + 핵심 이모지 교체
   - 헤더/CTA/대시보드 액션 아이콘 등 **navigational** 위치만 교체
   - 카테고리 이모지(📋💼🏆…)는 정체성이 있으므로 **유지** (혹은 컬러 라인 아이콘으로 통일하되 카테고리 의미는 보존)
   - 별점은 SVG 별로 교체

#### 산출물
- `src/components/ui/Card.jsx`
- 업데이트된 `src/index.css` 토큰
- 헤더/대시보드 헤더/주요 버튼 아이콘 교체

#### 완료 기준
- 모든 카드가 동일한 그림자·보더·hover 동작
- 이모지가 navigational 위치에서 사라짐
- 카테고리 컬러가 BlockCard에 시각적으로 반영됨

### P2. 자소서 저장 인프라 + 이력 페이지 (6~8h)

가장 큰 기능적 페인. 완성되면 서비스 톤이 "데모"에서 "도구"로 바뀐다.

#### DB 스키마

```sql
-- migrations/00X_cover_letters.sql
CREATE TABLE cover_letters (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id uuid NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  company text,            -- jobInfo.company
  position text,           -- jobInfo.position
  job_posting_raw text,    -- 원본 공고 텍스트 (재분석용)
  job_info jsonb,          -- 파싱 결과 (keywords, requirements 등)
  questions jsonb,         -- [{text, charLimit}]
  match_result jsonb,      -- matchBlocksToQuestions 결과
  status text DEFAULT 'draft', -- draft | completed | archived
  created_at timestamptz DEFAULT now(),
  updated_at timestamptz DEFAULT now()
);

CREATE TABLE cover_letter_answers (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  cover_letter_id uuid NOT NULL REFERENCES cover_letters(id) ON DELETE CASCADE,
  question_index int NOT NULL,
  block_id uuid REFERENCES experience_blocks(id) ON DELETE SET NULL,
  answer text,
  used_keywords text[],
  char_count int,
  generation_options jsonb, -- { tone: 'confident', ... } — P3에서 사용
  created_at timestamptz DEFAULT now(),
  updated_at timestamptz DEFAULT now(),
  UNIQUE (cover_letter_id, question_index)
);

-- RLS: user_id 기반 본인 데이터만
```

#### 페이지 변경

- **`/cover-letter`** → 이력 리스트 (회사별 자소서 카드)
  - 빈 상태일 때만 기존 입력 폼으로 다이렉트
- **`/cover-letter/new`** → 기존 입력/매칭/결과 흐름 (단, 단계마다 자동 저장)
- **`/cover-letter/:id`** → 저장된 자소서 작업 화면 (이어서 편집)

#### 클라이언트 변경
- `CoverLetterPage`의 `useState` 기반 상태를 `cover_letter` row 기반으로 전환
- 답변 수정 시 디바운스 후 `cover_letter_answers` upsert
- 공고 분석 직후 `cover_letters` row 즉시 생성 (status: draft)

#### 완료 기준
- 작업 도중 새로고침 → 이어서 편집 가능
- `/cover-letter`에서 회사별 이력 확인 + 재진입 가능
- 삭제/아카이브 가능

### P3. 자소서 재생성 + 톤 조정 + 블록 수동 매칭 (4~6h)

P2의 데이터 모델 위에서 동작.

#### 1) 답변 재생성
- `QuestionCard`에 "다시 생성" 버튼 추가
- 기존 답변을 덮어쓰는 대신 `cover_letter_answers`에 새 행을 만들고 최신만 노출 (이전 버전 1~2개 토글로 복구 가능)
- 사용량 카운터: 재생성도 `answer_gen` 호출이므로 한도 차감 — 무료 플랜에 명시 필요

#### 2) 톤/스타일 옵션
- 재생성 버튼 옆에 `Select` 또는 chip:
  - 톤: `기본` / `적극적` / `겸손한` / `데이터 중심`
  - 강조: `STAR 균형` / `결과 강조` / `과정 강조`
- `coverLetterEngine.generateAnswer`에 `options` 파라미터 추가 → 시스템 프롬프트에 합성
- 선택값은 `generation_options` jsonb에 저장 (재생성 시 디폴트로 복원)

#### 3) 블록 수동 매칭 변경
- `QuestionCard`에 "블록 변경" 드롭다운 (블록 검색 가능, 카테고리 필터)
- 변경 시 `cover_letter_answers.block_id` 업데이트, 기존 답변은 보존하되 "이 답변은 이전 블록 기준" 표식
- 자동 재생성은 하지 않음 (사용자가 명시적으로 "다시 생성" 눌러야 — 비용 보호)

#### 완료 기준
- 한 문항 = 다수 답변 시도 가능, 이전 시도 복구 가능
- 톤 옵션이 결과물 톤을 실제로 변경시킴 (수동 검증)
- 블록 교체 후 재생성 흐름이 직관적

### P4. 대시보드 정보 위계 개편 (6~8h)

#### 현재 문제
- 모든 섹션이 동등 비중 → "내가 뭘 봐야 하지?" 상태
- 블록 카드가 클릭 expand → 다 펼치면 정보 폭발

#### 새 구조

```
┌─────────────────────────────────────────────┐
│ [Hero zone]                                 │
│  • 역량 레이더 차트 (Recharts)               │
│  • "다음에 만들면 좋은 카테고리" 추천 카드    │
│  • 이번 달 사용량 progress (블록 N/3)        │
├─────────────────────────────────────────────┤
│ [Notion-style: 좌 리스트 / 우 미리보기]      │
│  좌 (1/3):                                  │
│    • 검색 + 카테고리 필터 + 정렬            │
│    • 블록 카드 (compact)                    │
│  우 (2/3):                                  │
│    • 선택된 블록 풀뷰                       │
│    • 인터뷰 이력, 수정/삭제                 │
├─────────────────────────────────────────────┤
│ [Collapsible] 업종/직무 추천                │
└─────────────────────────────────────────────┘
```

- 기본 선택: 가장 최근 블록
- 모바일: 좌/우 패널이 화면 단위로 분리 (현재 expand 토글 유지)
- 블록 0개 시: 빈 상태 그대로 유지

#### 완료 기준
- 데스크탑에서 "내 블록 훑어보기"가 클릭 토글 없이 가능
- 검색/필터로 10개+ 블록도 빠르게 탐색
- 역량 그래프가 첫 화면에 보이고, 부족한 역량 → 액션(블록 만들기) 직결

### P5. 랜딩 페이지 데모/시각화 (4~6h)

#### 추가/교체할 섹션
1. **Hero 직하 데모**: 인터뷰 → 블록 변환을 보여주는 짧은 인터랙티브 (혹은 자동 재생 애니메이션)
   - 가장 가벼운 구현: 미리 만든 대화 로그를 단계별 reveal + 마지막에 블록 카드 등장
2. **Before/After 비교**: "평범한 자기소개 한 줄 ↔ STAR 블록" 좌/우 카드
3. **소셜 프루프 자리**: "지금까지 N개의 블록이 만들어졌어요" (DB count 쿼리, 캐싱)
4. 기존 텍스트 섹션은 축약

#### 완료 기준
- 첫 스크롤 안에 "이 도구가 뭘 해주는지" 시각적으로 이해 가능
- 가입 전환 퍼널 측정 준비 (P6 분석 도입 시 활용)

### P6. 보강 묶음 (3~4h)

- **블록 검색/필터/정렬** (이미 P4에서 일부 구현되었을 가능성 — 잔여만 처리)
- **인터뷰 중단 버튼** — 현재 카테고리 선택 화면으로 돌아가기
- **업종 추천 백그라운드 실패 재시도** — 대시보드 진입 시 `recommended_industries === {}`인 블록 lazy 재호출
- **블록 강도 정렬 옵션**

## 4. 위험 요소 및 결정 사항

### 확정된 결정 (2026-04-26)
1. **자소서 재생성 사용량 정책**: **문항당 3회까지 무료** (그 이후 호출은 차감). `cover_letter_answers` 행 수 또는 별도 카운터로 문항 단위 트래킹. Edge Function `gemini-proxy`의 `answer_gen` 액션에서 문항 단위 카운트 확인 후 한도 초과 시에만 월별 카운터 증가.
2. **카테고리 컬러 매핑**: **4~5개 그룹 컬러**.
   - 🟦 학업/연구계 (팀프로젝트, 자격증/교육)
   - 🟩 실무계 (인턴, 아르바이트)
   - 🟧 도전계 (공모전, 개인프로젝트)
   - 🟪 대외활동계 (동아리, 해외경험)
   - (필요 시 기타 1색)
   - 정확한 카테고리 ID ↔ 그룹 매핑은 P1 착수 시 `config/categories` 확인 후 확정.
3. **랜딩 데모 방식**: **자동 재생 애니메이션**. 미리 만든 대화 로그를 단계별 fade-in → 마지막에 블록 카드 등장. 비로그인 인터랙티브 데모는 별도 기획으로 보류 (보안/비용 이슈).

### 위험
- **P2 마이그레이션**: 프로덕션 DB에 새 테이블 추가 — RLS 정책 검증 필수, 기존 사용자 영향 없음 확인.
- **P4 대시보드 Notion 패널**: 모바일 UX 검증 필요. 컴팩트 카드와 풀뷰의 정보 양 균형.
